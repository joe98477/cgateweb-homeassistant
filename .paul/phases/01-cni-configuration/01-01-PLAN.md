<objective>
## Goal
Add CNI (C-Bus Network Interface) IP address configuration for managed mode, so C-Gate can connect to the user's physical C-Bus network hardware (CNI, Wiser, etc.).

## Purpose
In managed mode, C-Gate runs locally inside the add-on but currently has NO way to know which CNI device to connect to. Without this, managed mode is non-functional for controlling C-Bus devices. This is the critical missing piece identified during project research.

## Output
- New `cni_ip` and `cni_network` config options in `config.yaml`
- A `cgate-setup` service that configures C-Gate's CNI connection after startup
- `project.start` added to C-GateConfig.txt so the project auto-starts with its networks
- Translations for new options
</objective>

## Research Context
CNI IP is project-level data in C-Gate, NOT a C-GateConfig.txt property.
Configuration requires C-Gate commands after the server starts:
  - `project use <PROJECT>`
  - `net create <NETWORK> cni <IP>`
  - `project save <PROJECT>`

`project.start=<PROJECT>` in C-GateConfig.txt makes C-Gate auto-start the project on launch.
Default CNI port is 10001 (implicit, not specified in net create).
`netcat-openbsd` is already installed in the Docker image.
</context>

  <name>Task 2: Add project.start to C-GateConfig.txt and create CNI setup service</name>
  <files>cgateweb/rootfs/etc/services.d/cgate/run, cgateweb/rootfs/etc/services.d/cgate-setup/run, cgateweb/rootfs/etc/cont-init.d/cgate-install.sh</files>
  <action>
    In cgate/run (services.d/cgate/run):
    - Add "project.start" to the SETTINGS associative array with value "${CGATE_PROJECT}"
    - This ensures C-Gate auto-starts the project and its configured networks on launch

    In cgate-install.sh:
    - When creating initial C-GateConfig.txt, also add `project.start=${CGATE_PROJECT}` alongside `project.default`

    Create new file cgate-setup/run (services.d/cgate-setup/run):
    - Shebang: #!/usr/bin/with-contenv bashio
    - Read CGATE_MODE, CNI_IP, CGATE_PORT, CGATE_PROJECT, CNI_NETWORK from bashio::config
    - If mode is not "managed" OR cni_ip is empty: exec sleep infinity
    - Wait for C-Gate to accept connections on 127.0.0.1:${CGATE_PORT} (poll with nc -z, max 30 attempts, 2s apart)
    - If C-Gate doesn't respond within 60s: log error and exec sleep infinity
    - Sleep 3 seconds for C-Gate to fully initialize
    - Send commands via netcat pipe to 127.0.0.1:${CGATE_PORT}:
      ```
      project use ${CGATE_PROJECT}
      net create ${CNI_NETWORK} cni ${CNI_IP}
      project save ${CGATE_PROJECT}
      quit
      ```
    - Use sleep 1 between each command in the pipe for C-Gate to process
    - Log success message
    - exec sleep infinity (service stays alive per s6 requirements)
    - Make the file executable (Dockerfile already has chmod for services.d/*/run)
  </action>
  <verify>
    - Verify cgate/run SETTINGS array includes project.start
    - Verify cgate-setup/run exists and is syntactically valid (bash -n check)
    - Verify cgate-install.sh writes project.start in initial config
  </verify>
  <done>AC-2, AC-3, AC-4 satisfied: C-Gate auto-starts project, CNI configured after startup, service idle when not needed</done>
